# Cross-Repository Notes for CompText AIR

This document tracks reusable concepts, architectural patterns, and interaction models from the connected CompText ecosystem. These are for conceptual alignment and design inspiration only; no code or runtime assumptions are imported.

## 1. Repository Inventory

| Repository | Primary Role | Status in AIR |
| :--- | :--- | :--- |

## 2. Useful Concepts for AIR

### Deterministic Context
- **Classification:** Candidate
- **Source:** `comptext-cli`
- **AIR Application:** Treat context as a versioned artifact. Ensure AIR schemas support deterministic rendering of context to avoid LLM drift.

### Evidence Package (.spkg)
- **Classification:** Candidate
- **Source:** `comptext-sparkctl`
- **AIR Application:** Use the `.spkg` structure as a reference for AIR's event containers and replay contracts.

### `--json` Command Contract
- **Classification:** Candidate
- **Source:** `comptext-cli`
- **AIR Application:** All AIR-compliant tools should provide a machine-readable `--json` output for agent introspection.

### Resource/Tool DSL
- **Classification:** Candidate
- **Source:** `comptext-dsl`
- **AIR Application:** Align `air.schema.json` with DSL resource/tool definitions to ensure future compatibility.

### Single-Char Command Syntax (Codex V5)
- **Classification:** Inspiration
- **Source:** `comptext-codex`
- **AIR Application:** Consider high-density mappings for high-frequency events in replay contracts to minimize token overhead.

### Integrity Chain
- **Classification:** Inspiration
- **Source:** `comptext-sparkctl`
- **AIR Application:** Linking hashes of AIR events to create an immutable audit trail.

## 3. Concepts Explicitly Not Imported

- **Provider Runtimes:** AIR does not implement LLM provider logic.
- **MCP Server Logic:** AIR defines the intermediate representation, not the transport layer or server implementation.
- **File System Mutation:** AIR focuses on representation and validation, not the execution of file system changes.
- **Credential Management:** Secrets and keys are strictly outside the scope of AIR.

## 4. Possible Phase 6 Candidates

1. **AIR-to-SPKG Transformer:** Tools to package AIR events into `.spkg` files for `sparkctl` consumption.
2. **Codex Syntax Support:** Formal mapping between AIR event types and `comptext-codex` single-character commands.
3. **DSL Manifest Validation:** Using AIR schemas to validate tool manifests generated by `comptext-dsl`.
4. **Adversarial Replay Simulation:** Using AIR fixtures to test the robustness of evidence verification in `sparkctl`.
5. **Context Checkpointing:** Schema support for capturing full context snapshots at specific event boundaries.

## 5. Boundary Warnings

- **No Production Claims:** These concepts do not imply AIR is production-ready or provides forensic assurance.
- **No MCP Claims:** Referencing `comptext-mcp` does not mean AIR supports MCP natively; it only aligns schemas.
- **Local-Only:** AIR operates under the assumption of a "Network Denied" local-first boundary.

## 6. No-Code-Import Statement

CompText AIR is a schema, fixture, and validation project. **No code, libraries, or binary dependencies are imported from the referenced repositories.** All implementations in AIR are original and follow the AIR-specific goals of representation and replay integrity.
